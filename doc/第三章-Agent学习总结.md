# 第三章 Agent 学习总结

## 1. 本章当前学习目标

这一章目前学的不是“高级算法 Agent”，而是 **AI 应用开发里最常见的 Agent 基础能力**。

当前阶段的目标是：

1. 理解 Agent 和普通聊天接口、RAG 的区别
2. 理解 Tool Calling 的基本原理
3. 能手写一个最小 Agent
4. 理解 Agent 的循环、多工具选择、会话记忆
5. 理解 Agent 和 RAG 如何结合
6. 为后续学习 LangChain / LangGraph 做准备

一句话概括：

**Agent = 大模型 + 工具 + 决策流程 + 上下文管理**

---

## 2. 当前已经学会的内容

### 2.1 Agent 和普通聊天、RAG 的区别

目前已经明确：

- 普通聊天接口：
  - 用户提问
  - 模型直接回答
- RAG：
  - 用户提问
  - 先检索知识库
  - 再把检索结果给模型回答
- Agent：
  - 用户提问
  - 模型先判断要不要调用工具
  - 如果要，就调用工具
  - 工具执行完后，再继续思考，最后回答

关键理解：

**RAG 更像一种“检索增强回答”的能力；Agent 更像一种“会判断、会行动、会继续决策”的能力。**

---

### 2.2 Tool Calling 基础原理

已经学会：

- 工具不是模型自己执行的
- 模型只负责：
  - 判断要不要调用工具
  - 决定调用哪个工具
  - 决定工具参数是什么
- 真正执行工具的是后端代码

当前项目中已经写过的工具有：

- `get_current_time`
- `get_mock_weather`
- `search_demo_knowledge`

关键理解：

**模型不直接执行 Python 函数，模型只是“发起工具调用请求”，后端才是真正执行者。**

---

### 2.3 最小 Tool Calling Demo

已经完成并测试：

- `POST /chat/tool-demo`

这个 demo 学会了完整的最小链路：

1. 用户提问
2. 把 `tools` 和 `messages` 发给模型
3. 模型返回 `tool_calls`
4. 后端执行工具
5. 把工具结果作为 `tool` 消息再发回模型
6. 模型生成最终回答

关键理解：

**Tool Calling 最重要的不是工具函数本身，而是“两次请求模型”的完整闭环。**

---

### 2.4 Agent Loop

已经完成并测试：

- `POST /chat/agent-loop-demo`

这一部分学会了：

- 不再固定只调用一次工具
- 模型可以在一个循环中连续多轮决定是否继续调用工具
- 通过 `max_steps` 控制最大循环次数，避免无限循环

关键理解：

**Agent Loop 的本质是：模型每一轮都重新判断“信息够不够，如果不够，下一步该调用什么工具”。**

---

### 2.5 会话记忆 Agent

已经完成并测试：

- `POST /chat/agent-session-demo`

这一部分学会了：

- 使用 `session_id` 区分不同会话
- 使用 `session_store` 保存当前会话历史
- 下次请求时，把历史消息恢复成 `messages`
- 让模型能理解省略表达

例如：

- 第一轮：“记住城市是杭州”
- 第二轮：“那现在告诉我要不要带伞”

模型依然能知道是在说杭州。

关键理解：

**所谓会话记忆，本质不是模型永久记住了，而是后端把历史消息保存下来，并在下次请求时重新带给模型。**

---

### 2.6 Agent + RAG

已经完成并测试：

- `POST /chat/agent-rag-demo`

这一部分学会了：

- 不再固定走 RAG
- 而是把 RAG 包装成一个工具 `search_demo_knowledge`
- 由 Agent 自己判断什么时候调用这个工具

当前已经实现：

- 问时间 -> 调时间工具
- 问天气 -> 调天气工具
- 问 FastAPI / 知识库内容 -> 调知识库检索工具

关键理解：

**这一节不是重新做 RAG，而是把 RAG 变成 Agent 的一个可选工具。**

---

### 2.7 多工具路由 + 缺参追问

已经完成并测试：

- `POST /chat/agent-route-demo`

这一部分学会了：

- Agent 不只是“会选工具”
- 还要先判断“参数是否完整”
- 参数不完整时，先追问，不要盲目调用工具

例如：

- 用户说：“帮我查一下天气”
  - 没有城市
  - Agent 应该先追问城市

- 用户说：“帮我查一下杭州天气”
  - 参数完整
  - Agent 才调用天气工具

关键理解：

**Agent 从“会调用工具”升级成“会先检查条件，再决定是否调用工具”。**

---

### 2.8 异常处理的思路

这一部分目前先学了思路，没有继续深写完整代码实现。

已经明确：

- 工具调用不只有成功路径，也有失败路径
- 工具可能因为参数错误、工具不支持、外部服务异常而失败
- 后续进入 LangChain 后，再结合框架做更完整的异常处理、兜底、fallback、retry

关键理解：

**Agent 的成功路径和失败路径都需要设计。**

## 4. 当前最重要的核心理解

这一章到目前为止，最关键的理解有下面几条：

### 4.1 Agent 不等于框架

LangChain / LangGraph 不是 Agent 的本质，它们只是对 Agent 逻辑的工程化封装。

当前手写部分学的是：

- Agent 的底层原理
- Tool Calling 的本质
- Loop / Memory / RAG / Routing 的逻辑

后面框架学的是：

- 怎么更高效地组织这些能力

---

### 4.2 Tool Calling 的本质是“模型决策 + 后端执行”

模型不会真的执行代码，它只负责：

- 选工具
- 给参数

后端负责：

- 真正执行工具
- 把结果回传给模型

---

### 4.3 记忆的本质是“保存历史并重新带回”

当前是存在字典 `session_store` 里；
真实业务里通常会存数据库 / Redis。

区别只是存储介质不同，主流程不变：

- 保存历史
- 下次再取出
- 拼进 `messages`

---

### 4.4 RAG 和 Agent 可以结合

RAG 不是只能单独作为一条固定链路；
它也可以作为 Agent 的一个工具，由模型自己决定是否调用。

---

### 4.5 很多 Agent 效果优化，先体现在 Prompt 上

例如：

- 缺参先追问
- 有参数再调用工具
- 优先使用知识库结果回答

这些很多时候不需要先大改代码，而是先写更明确的行为约束 Prompt。

---

## 5. 学习中遇到的关键问题

这一章学习过程中，已经遇到并解决过很多很有价值的问题。

### 5.1 `tool_choice="auto"` 是什么意思

已经理解：

- 让模型自己判断是否调用工具
- 不是强制调用

---

### 5.2 第二次请求为什么还要把 `assistant` 和 `tool` 消息带回去

已经理解：

- 第二次请求不是重新开始
- 必须把：
  - 用户原问题
  - 模型刚才的工具调用决定
  - 工具执行结果
  一起带给模型

这样模型才能基于完整上下文生成最终回答。

---

### 5.3 `max_steps` 不是聊天轮数

已经理解：

- `max_steps` 控制的是 Agent 内部最多进行多少轮工具决策循环
- 不是用户和模型的聊天轮数

---

### 5.4 `session_store` 为什么能直接取值

已经理解：

- 因为用了 `defaultdict(list)`
- key 不存在时会自动创建空列表

---

### 5.5 为什么 `tool-demo` 和 `agent-route-demo` 行为不同

已经理解：

- `tool-demo` 更像基础演示版，主要展示调用链路
- `agent-route-demo` 更像业务规则版，重点是缺参追问和参数完整性判断
- 这一节主要是 Prompt 约束升级，不是代码结构大改
