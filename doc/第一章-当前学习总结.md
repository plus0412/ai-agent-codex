## 3. 核心知识点总结

### 3.1 项目分层结构

当前项目的核心分层：

1. `main.py`
   - 项目入口
   - 创建 FastAPI 应用
   - 注册路由与异常处理
2. `routers`
   - 放接口定义
   - 决定“哪个路径对应哪个函数”
3. `schemas`
   - 放请求体、响应体的数据结构
   - 相当于 DTO / VO
4. `services`
   - 放业务逻辑
   - 路由层不要把所有逻辑都写完
5. `config.py`
   - 放配置读取逻辑
   - 管理 API Key、模型名、base_url 等
6. `exceptions.py`
   - 放统一异常处理

核心理解：

- `router` 负责接请求
- `service` 负责处理逻辑
- `schema` 负责约束输入输出
- `config` 负责统一配置

---

### 3.2 FastAPI 基础

已经掌握的基础点：

1. `FastAPI()` 用来创建应用
2. `APIRouter()` 用来组织一组接口
3. `@router.get(...)` / `@router.post(...)`
   - 可类比 Java 的 `@GetMapping` / `@PostMapping`
4. `prefix`
   - 给一组接口统一加路径前缀
5. `response_model`
   - 用 schema 约束响应结构

核心理解：

- 路径不是全部，接口还包含请求方式
- 一个 router 里可以放多个接口

---

### 3.3 Pydantic 与 Schema

已经掌握：

1. `BaseModel`
   - 用来定义结构化数据
2. `Field(...)`
   - 给字段加校验规则与文档说明
3. schema 既可以约束请求，也可以约束响应
4. schema 在结构化输出中还能作为模型输出格式模板

当前理解可类比 Java：

- schema ≈ DTO / VO
- `Field(...)` ≈ 字段约束 + 文档说明

---

### 3.4 请求方式与参数传递

已经掌握：

1. `GET`
   - 更常用于查询
2. `POST`
   - 更常用于提交数据
3. 查询参数
   - 写在 URL 后面
4. 请求体
   - 放在请求内部，常见为 JSON

核心理解：

- 浏览器地址栏默认更适合测 `GET`
- `POST` 更适合 AI 应用里的复杂输入

---

### 3.5 配置管理

已经掌握：

1. `.env.example`
   - 配置模板
2. `.env`
   - 真实运行配置
3. `load_dotenv()`
   - 加载 `.env`
4. `os.getenv(...)`
   - 读取环境变量

核心理解：

- `.env.example` 可以写安全的示例值
- `.env` 才是程序真正读取的本地配置
- API Key 不能写死在代码中

---

### 3.6 真实模型调用

已经掌握：

1. 通过阿里云百炼官方兼容接口调用模型
2. 模型配置包括：
   - `DASHSCOPE_API_KEY`
   - `LLM_BASE_URL`
   - `LLM_MODEL`
3. `messages` 是发给模型的消息列表
4. `system / user / assistant` 是三种核心角色

当前核心链路：

1. 接收用户请求
2. 组装 `messages`
3. 调用模型 API
4. 从返回结果中提取回答文本
5. 再包装成响应对象返回

---

### 3.7 模型返回结果理解

已经掌握：

1. `completion`
   - 是模型完整返回对象，不是单个字符串
2. `completion.choices[0].message.content`
   - 用于获取普通文本回答
3. `or ""`
   - 兜底空字符串

核心理解：

- `choices` 是候选结果列表
- `[0]` 表示取第一个结果
- `message.content` 才是真正回答文本

---

### 3.8 结构化输出

已经掌握：

1. `response_format=ChatSummaryResponse`
   - 让模型按指定结构返回
2. `parse(...)`
   - 用于结构化解析
3. `message.parsed`
   - 表示 SDK 已经解析好的结构化结果对象
4. `message.content`
   - 表示原始文本内容

核心理解：

- `content` 是原始文本
- `parsed` 是解析后的结构化对象
- 结构化输出非常适合做业务型 AI 接口

---

### 3.9 流式输出

已经掌握：

1. `stream=True`
   - 开启流式返回
2. `for chunk in stream`
   - 一段段读取模型输出
3. `delta.content`
   - 当前这一小段新增内容
4. `yield`
   - 一段段往外返回
5. `StreamingResponse`
   - FastAPI 的流式响应对象

核心理解：

- 流式返回的是增量，不是覆盖
- 每次来的都是新增片段
- 最终完整回答是由多个片段累积拼接得到的

---

### 3.10 多轮对话与会话管理

已经掌握：

1. `session_id`
   - 区分不同对话
2. `session_store`
   - 保存会话历史
3. `ChatTurn`
   - 表示一条历史消息
4. `build_session_reply(...)`
   - 根据历史构造多轮 messages

核心理解：

- 模型不会自动记住上下文
- 所谓“记住历史”，本质是后端把历史消息再次发给模型
- history 中必须同时保存 user 和 assistant

---

### 3.11 异常处理

已经掌握：

1. `ValueError`
   - 用于主动抛业务错误
2. `@app.exception_handler(...)`
   - 注册统一异常处理
3. `JSONResponse`
   - 手动构造错误返回

核心理解：

- 统一异常处理很像 Java 的全局异常处理
- 出错时也要返回结构化响应

---

### 3.12 异步基础

已经掌握初步认识：

1. `async def`
   - 异步函数
2. 适合 IO 等待型场景
3. 在 FastAPI 与 AI 应用中非常常见

当前只要求：

- 看到 `async` 不慌
- 知道它和普通 `def` 不一样
