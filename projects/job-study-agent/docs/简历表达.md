# JobStudyAgent 简历表达

## 1. 项目一句话介绍

基于 `FastAPI + LangChain + LangGraph + RAG` 实现了一个面向求职与学习场景的任务型 AI Agent，支持多轮会话、资料上传建库、知识库检索、岗位分析与学习计划生成。

---

## 2. 简历项目描述写法

### 版本一：偏项目概述

独立设计并实现 JobStudyAgent 项目，围绕“求职资料理解 + 学习建议生成”场景，完成 Agent 主流程编排、知识库检索、任务型节点拆分和多轮会话记忆。

### 版本二：偏技术实现

基于 FastAPI 搭建 AI 应用后端，使用 LangGraph 编排 `route_intent / retrieve_knowledge / job_analysis / study_plan / save_session` 等节点，结合本地 RAG 知识库实现资料问答与任务型输出。

---

## 3. 简历亮点 bullet

- 基于 `FastAPI + LangChain + LangGraph` 实现任务型 Agent，支持普通问答、知识库问答、岗位分析和学习计划生成等多类场景
- 设计并实现本地 RAG 能力，支持资料上传建库、文本切分、Embedding 检索、知识片段召回与基于知识回答
- 使用 LangGraph 对 Agent 主流程进行状态编排，完成意图路由、上下文检查、知识检索、任务节点执行与会话保存
- 实现内存版多轮会话记忆与调试信息返回，支持查看图执行步骤、当前意图、知识片段命中情况等运行信息
- 项目采用 `router / schema / service / graph / storage` 分层结构，便于后续扩展 MySQL、Redis、前端页面和多 Agent 协作

## 4. 面试时可以怎么讲

### 4.1 先讲业务场景

这个项目解决的是一个真实问题：用户在求职和学习过程中，会有岗位 JD、学习笔记、项目总结等资料，希望系统能理解这些资料，并进一步给出岗位分析和学习建议，而不只是普通聊天。

### 4.2 再讲系统结构

我把项目拆成两层：

- 第一层是知识库能力，包括上传资料、切分、Embedding、检索
- 第二层是 Agent 能力，包括意图识别、知识检索分支、岗位分析节点、学习计划节点和会话记忆

### 4.3 再讲技术亮点

- 主流程不是简单 if/else，而是用 LangGraph 编排成图
- 不同问题会进入不同节点，例如普通聊天走 `direct_answer`，资料问答走 `retrieve_knowledge -> answer_with_knowledge`，岗位分析走 `retrieve_knowledge -> job_analysis`
- 为了方便调试和展示，还补了 debug 返回结构，可以直接看到图执行步骤和检索命中情况

### 4.4 最后讲可扩展点

后续还可以继续扩展：

- 接 MySQL 持久化会话
- 接 Redis 做缓存或短期记忆
- 增加前端页面
- 增加更多任务型节点
- 增加多 Agent 协作

## 5. 面试高频问题回答方向

### 5.1 为什么要用 LangGraph

因为项目不只是调用模型，还包含意图路由、上下文检查、知识检索、任务节点执行和兜底逻辑。用图来表达这些流程，比把所有逻辑塞进一个超长 service 更清晰、更可维护。

### 5.2 为什么不是单纯用 LangChain

LangChain 更偏提示词、消息对象和模型封装；LangGraph 更适合表达“多步骤、有状态、有分支”的业务流程。这个项目同时用了两者，但流程编排核心放在 LangGraph。

### 5.3 这个项目最有代表性的能力是什么

最有代表性的能力不是“会调用模型”，而是“系统会判断当前是什么任务，并进入不同节点完成不同处理”。
